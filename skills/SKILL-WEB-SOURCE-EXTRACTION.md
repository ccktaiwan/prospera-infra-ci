## 1. 問題緣起（Q1 Proposal 沉澱）

品牌交付物必須填入客戶真實資料，但客戶 SSOT（BrandKB JSON）有時資料不完整。
開始用官方網站補充時，出現三類失效：

| 失效類型 | 案例 | 後果 |
|---|---|---|
| **人名轉寫錯誤** | 從英文拼音回推中文人名（楊依群 → 正確應為 楊逸群）| 人名錯誤寫入交付物 |
| **單一來源直接寫入** | 只看官網 about-us 頁，未交叉驗證法人資訊 | 若官網有誤，無法識別 |
| **混合標記混入** | 推測或推導內容被標記為事實（「院長（AI推測）」）| 假事實進入 SSOT 衍生物 |

本 Skill 定義從官方網站蒐集資料、交叉驗證、寫入交付物的完整紀律。

---

## 2. 適用場景

- BrandKB SSOT 缺乏客戶資訊，需從官網補充
- 需要驗證客戶法人資訊（公司全名/負責人/地址/電話/統編）
- 任何「官方網站資料 → 交付物」的寫入流程

**不適用**：純 BrandKB JSON 內部資料讀取（走 SKILL-13）

---

## 3. 來源分級（裁定1）

| Tier | 來源類型 | 範例 | 用途 |
|---|---|---|---|
| **Tier 0** | 設立登記（經濟部商業司） | 公司名稱/負責人/統編/地址 官方登記 | **法律權威**：法人資訊最終真相源 |
| **Tier 1** | 官網、Google 商家、Meta 粉專、LINE@ | luckyphoenixmed.com、Google Maps 頁面 | 一般事實：單一來源 + URL 即可 |

**寫入規則**：

```
一般事實（非法人核心欄位）：
  → 單一 Tier 1 來源 + URL 即可寫入
  → 標記格式：（來源：URL，日期）

關鍵事實（公司全名/負責人/地址/電話/統編）：
  → 至少兩個官方來源交叉比對，一致才寫入
  → 權威序：Tier 0 設立登記 > 官網 > Google商家/Meta/LINE@
  → 不一致時：以較高 Tier 者為準，標記差異

無法取得兩源：
  → 標「⚠️待補（需交叉驗證）」，不寫入單一來源推測值
```

## 4. 三層真實性（裁定3）

| 層 | 定義 | 可寫入交付物？ |
|---|---|---|
| **第一層：公開事實** | 公司登記/官網/Google商家確認的組織事實 | ✅ 可（標來源） |
| **第二層：機構核可內容** | 官網 news/具名文章/品牌公告（院長具名發言） | ✅ 可（逐字引用，標 URL）|
| **第三層：個人私域** | 個人信念/內心聲音/個人語氣/未發表觀點 | 🚫 禁止生成 |

**保守原則**（裁定3）：
```
分不清第二層/第三層 → 當第三層（標待補、不寫入）
分不清事實/推測     → 當推測（不寫入）
原則：寧可漏寫真的，不可寫進假的。不做判斷樹。
```

---

## 5. 四步抽取法（官網資料版）

```
Step 1｜確認官網來源頁面
  - 記錄完整 URL
  - 確認是否為中文原頁（有中文原頁必用中文原頁）
  - 確認頁面類型：about/team（第一層）或 news/具名文章（第二層）

Step 2｜逐字擷取（不轉寫不重構）
  - 人名：照抄中文原頁，禁止從拼音或英文回推中文名
  - 若只有拼音：先查中文原頁確認，不得自行轉寫
  - 具名引言：逐字照抄，不改寫，不摘要

Step 3｜關鍵事實交叉驗證
  - 法人資訊（公司全名/負責人/地址/電話/統編）：
    → 同時查 Tier 0 設立登記 + Tier 1 官網
    → 兩源一致才寫入；不一致→標記差異，以 Tier 0 為準

Step 4｜標記層次 + 更新 SOURCES.md
  - 每個寫入欄位標明：（來源：URL，日期，Tier）
  - 無來源欄位：標「⚠️待補」
  - 更新 SOURCES.md 的「官網公開資料來源」段
```

## 6. 人名規則（並入自 SKILL-13 實戰沉澱）

```
✅ 照抄官網中文原頁的人名
✅ 若官網有中英雙版，取中文版
✅ 若只有拼音/英文 → 先查中文版再寫
🚫 禁止：從拼音重構中文名（會出現同音字錯誤，如楊依群→楊逸群）
🚫 禁止：從英文名推測中文名
🚫 禁止：自行音譯或造字
```

**案例（GATE 4 發現）**：
- 楊依群（錯，從拼音 Yang Yi-Qun 重構）→ 楊逸群（正確，照抄官網）

## 7. 混合標記規則（裁定2）

| 標記類型 | 定義 | 處置 |
|---|---|---|
| **純待補** | `⚠️待補`（無來源，已知缺乏）| ✅ 合法狀態，不 BLOCK |
| **混合生成標記** | 含「AI推測/AI生成/可能是/AI推導」| 🚫 **BLOCK** — 逼純待補或補真來源 |
| **來源標記** | `（來源：URL）` | ✅ 合法狀態 |

**為什麼 BLOCK 混合標記**：混合標記表示 AI 生成的內容混入了事實欄位，
形成假事實（False Grounding）。一旦進入 SSOT 衍生物，後續 AI 會引用為「已知事實」，
難以清除。早期 BLOCK 的成本遠低於後期清除成本。

**Fitness Function 自動偵測**：`fitness_ssot_source_check.py`（Fitness C，`prospera-constitution-governance/governance/fitness/`）

## 9. known_failures 對應

| 症狀 | 修法 |
|---|---|
| 人名從拼音重構（如楊依群）| 回到官網中文原頁確認，照抄，不自行轉寫 |
| 法人資訊只有一個來源 | 查設立登記 + 官網交叉驗證，一致才寫入 |
| 欄位標「可能是」「AI推測」| 改為「⚠️待補」或補充真實來源 |
| 個人聲音/語氣被生成 | 降至第三層，全段標「⚠️待補（需具名文章/官方影片）」|
| 官網無中文版只有拼音 | 先查中文原頁；真的無中文版才保留拼音，標「⚠️無中文原頁，待確認」|
